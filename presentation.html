01 · 主題名稱與系統目的

我們要解決的
兩個揪團痛點

校園外送費高、揪團過程混亂——FoodWeb 用私人揪團房間與透明申請機制，一次解決。
📦
外送費太貴，一個人點不划算
系辦、實驗室、宿舍各自訂餐，同一棟大樓多次外送， 每人額外負擔 40–80 元運費，長期累積浪費可觀。
🤝
揪團流程不透明，房主難以彙整
以往透過通訊軟體揪團，點餐資訊散落各處， 房主難以彙整訂單，成員無法即時掌握累積金額與結單進度。
→
FoodWeb 提供即時校園揪團平台，以 私人揪團房間 與 申請加入審核 機制， 讓中興大學師生透明分攤運費、輕鬆統一外送。
02 · 使用情境說明

誰會在什麼時候
使用 FoodWeb？

兩種最常見的校園情境，對應「發起揪團」與「加入揪團」兩條路徑。
01
發起揪團
午餐時間，實驗室同學想一起訂外送
12:00，研究生小明想揪人訂麥當勞， 填入餐廳連結、取餐地點、外送運費與結單時間， 系統自動建立私人揪團房間， 分享連結給同學加入，房主逐一審核申請， 金額累積達標後統一下單，省下人均 45 元運費。
使用者：在校學生 / 研究生 / 教職員（房主）
02
加入揪團
看到揪團通知，想加入並填寫點餐
12:05，佳慧在首頁看到「胡椒廚房」揪團正在進行， 點擊進入揪團房間，填寫點餐內容與金額後送出申請， 系統顯示目前累積金額進度條與成員清單， 房主接受後申請即確認，等待結單取餐。
使用者：在校學生 / 研究生 / 教職員（成員）
03 · 主要功能規劃

六大核心功能

涵蓋帳號管理、揪團全流程與即時追蹤，共 6 個已實作功能模組。
F-01
👤
會員帳號系統
支援註冊、登入、登出，個人頁面可修改姓名、Email、大頭照，並提供刪除帳號功能。帳號資料以 UUID 管理。
帳號管理
F-02
🍜
揪團外送發布
填入揪團名稱、外送連結、取餐地點、外送運費與結單時間，系統自動建立獨立揪團房間並開放申請加入。
揪團外送
F-03
🔍
列表瀏覽與篩選
首頁即時顯示所有揪團，支援依餐廳類別（速食、中式、飲料、日式、其他）篩選，快速找到目標揪團。
揪團外送
F-04
📊
揪團房間進度追蹤
揪團房間顯示目前累積金額進度條、已接受成員清單與倒數結單計時器，所有人即時掌握揪單狀態。
揪團外送
F-05
✅
申請加入審核
成員填寫點餐內容與金額後送出申請（status=0），房主可接受（1）或婉拒（2），成員也可自行取消（3）。
加入管理
F-06
📋
個人資料管理
個人頁面顯示揪團參與紀錄與誠信積分，可修改暱稱、大頭照與密碼，所有變更即時同步至資料庫。
帳號管理

05 · 操作流程圖

使用者操作流程

從進入網站到完成主要任務的完整路徑。
🌐 進入 FoodWeb
foodweb.nchu.edu.tw
↓
選擇功能模組
Hero 區點選 Tab
↓
🍜 發起揪團路徑
1
點擊「發布新揪團」，填寫揪團名稱、外送連結、地點與運費
2
送出後 API 建立揪團房間（POST /api/team），首頁即時列出
3
等待成員送出加入申請，在房間內逐一接受（status=1）或婉拒（status=2）
4
結單時間到，房主彙整訂單統一下單取餐
✓ 完成揪團，省下運費
🚀 加入揪團路徑
1
首頁瀏覽揪單列表，依分類篩選或直接查看進度條
2
點擊「進入揪團房間」，填寫點餐內容與金額送出申請（POST /api/team/{id}/join）
3
等待房主審核，確認後 status 更新為 1（已接受），加入成員清單
4
若需取消可自行撤回申請（PATCH status=3），結單前均可操作
✓ 申請通過，等待結單取餐

python-dotenv
環境變數管理
main.py
FastAPI App 入口 ＋ Uvicorn 啟動
建立 FastAPI app 實例，透過 uvicorn 以 ASGI 方式啟動伺服器。 定義所有 RESTful API 路由，依資源分為帳號（signup / login / logout / 查詢 / 修改 / 刪除）、揪團（建立 / 列表）、加入申請（提交 / 審核）。 使用 Depends(get_db) 注入資料庫 Session。
POST /api/account/signup POST /api/account/login GET /api/team POST /api/team POST /api/team/{id}/join PATCH /api/team/{id}/join/{uid}
database.py
資料庫連線設定
以 python-dotenv 從 .env 讀取 DATABASE_URL， 使用 SQLAlchemy create_engine 建立 PostgreSQL 連線引擎， 配置 SessionLocal（autocommit=False, autoflush=False）確保手動 commit 控制， 並輸出 Base 供所有 ORM 模型繼承。
models.py
SQLAlchemy ORM 資料表模型
定義三張資料表的 Python 類別，對應資料庫實際欄位：
User（uid PK, name, mail unique, hashed_password）
Team（orderId PK, title, url, location, deliverFee, endAt, ownerId FK→User）
JoinTeam（orderId+uid 複合 PK, foodName, price, status 預設 0）
schemas.py
Pydantic 資料驗證 Schema
定義每個 API 端點的 Request Body 與 Response Model， 確保型別安全並自動生成 Swagger 文件。共 10 個 Schema 類別，包含： UserSignupCreate / UserLoginResponse / GetUser / UserChange、 TeamCreate / TeamResponse（含 member 清單與 totalPrice）、 JoinTeamCreate / HandleJoinTeam 等。
